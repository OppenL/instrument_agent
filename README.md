# 多智能体仪表控制设计说明

> 适用于：仪表控制 / 实验室自动化 / 数据隐私敏感场景

---

## 1. 系统概述

### 1.1 核心能力

用户通过自然语言描述测量需求，系统自动：
1. 理解用户意图
2. 确定需要的仪器
3. 指导连线
4. 检查仪器状态
5. 配置参数
6. 执行测量
7. 返回结果

### 1.2 硬件基础

| 组件 | 规格 |
|------|------|
| CPU | AMD 锐龙 AI MAX+ 395（16核 Zen5） |
| GPU | Radeon 8060S（支持 ROCm） |
| NPU | 锐龙 AI 专用 NPU |
| 内存 | 128GB DDR5 |
| 存储 | 2TB NVMe SSD |
| 显存 | 统一内存架构（UMA），最高 96GB |

### 1.3 分工原则

| 计算单元 | 任务 |
|---------|------|
| CPU | 任务调度、仪表通信、数据处理 |
| GPU | LLM 推理、Embedding、图像理解 |
| NPU | 持续在线任务（预留） |

## 2. 系统架构

### 2.1 多 Agent 架构的价值

**多 Agent 架构的价值：**

| 场景 | 说明 |
|------|------|
| **动态组合** | 不同测量任务需要不同的仪器组合，LLM 按需选择调用 |
| **知识隔离** | 每个 Specialist 专注一个仪器领域，便于管理和扩展 |
| **独立演进** | 各仪器驱动独立开发，互不影响 |
| **并行执行** | 无依赖的 Agent 可并行调用，提高效率 |
| **易于调试** | 问题定位到具体 Agent，便于排查 |

**示例：**

```
任务 1："测 S 参数"
→ 只需 VNA Agent

任务 2："做 MIMO 通道仿真"
→ 需要 VNA + Switch + CE + APM Agent

任务 3："单通道测量"
→ 只需 VNA Agent
```

LLM 根据任务类型动态决定调用哪些 Agent，不是每次都全量执行。

**对比单 Agent：**
- 单 Agent 需要在代码里写 if-else 判断调用哪个驱动
- 多 Agent 让 LLM 自主决策，更灵活

### 2.2 Specialist Agents

| Agent | 职责 | 驱动调用 |
|-------|------|---------|
| VNA Specialist | 测量 S 参数、注入信号、获取热力图 | vna_driver |
| CE Specialist | 配置通道仿真器（多径、时延、衰减） | ce_driver |
| APM Specialist | 配置幅度、相位 | apm_driver |
| Switch Specialist | 路由切换、端口映射 | switch_driver |
| Analyzer Specialist | IFFT/FFT/波束赋形/频谱分析 | numpy |

**注意**：所有 Specialist **并行**调用，由 Coordinator 按需调度。

### 3.3 记忆模块

**目的**：保存每次实验的连线配置和仪器参数，关闭应用再打开不丢失。

**数据库表结构**

```sql
-- 实验配置表
CREATE TABLE experiments (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100),           -- 实验名称
    created_at DATETIME,        -- 创建时间
    updated_at DATETIME,        -- 更新时间
    wiring JSON,               -- 连线配置
    instruments JSON,           -- 仪器配置
    status VARCHAR(50),         -- running/completed/aborted
    notes TEXT
);
```

**自动保存机制**

```
每次配置变更 → 自动保存到数据库 → 不需要用户手动保存
```

**启动恢复流程**

```
用户打开应用 → Memory 模块查询最近实验
                    │
                    ▼
        [继续上次的实验] 或 [新建实验]
```

**模块关系**

```
Coordinator Agent（调度） ←→ Memory 模块（存储）
     负责任务执行              负责配置记忆
```

## 5. Agent 检查逻辑

### 5.1 各 Agent 自检

| Agent | 检查命令 | 验证内容 |
|-------|---------|---------|
| VNA | `*IDN?` + `CAL?` | 设备身份 + 校准状态 |
| Switch | `*IDN?` + 路由查询 | 设备身份 + 端口可达 |
| CE | `*IDN?` + `CHAN:STAT?` | 设备身份 + 通道状态 |
| APM | `*IDN?` + `AMP?`/`PHA?` | 设备身份 + 当前设置 |

### 5.2 异常处理

| 异常 | 处理方式 |
|------|---------|
| 仪表连接失败 | 提示用户检查连接 |
| 配置失败 | 重试或报错 |
| 测量失败 | 记录错误，继续下一项 |

| BGE-M3 | Embedding（dense+sparse） | FP16 | ~1GB |

### 6.2 显存分配

| 组件 | 显存 |
|------|------|

| Qwen2.5 72B | 42GB |
| llava:13b | 13GB |

| **合计** | ~96GB |

## 10. 核心原则

1. **无 MCP 层**：Agent 直接 import 驱动，不走 MCP 协议
2. **本地优先**：所有数据本地存储，不依赖外部服务
3. **按需调用**：Coordinator 根据需求决定调用哪些 Agent，不是全量执行
4. **知识隔离**：不同仪器类型、型号的知识库隔离存储
